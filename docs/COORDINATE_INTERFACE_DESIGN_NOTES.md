**來源**:Dobot 官方應用手冊《MG400 实训台视觉定位抓取码垛说明》(V1.1, 2023-01-03, 20 頁)+ 本專案對話中的萃取與推導。
**性質**:**設計筆記 / 二手萃取,非官方逐字 spec**。供開發 agent / Claude Code 設計 MG400 坐標系介面時參考其「概念與架構」,不作為協議權威。
**用途**:把視覺定位(搭配本專案 CV / ArUco)所需的坐標系觀念,沉澱成介面設計準則。
**官方權威來源**(設計細節以這兩份為準):[`OFFICIAL_COORDINATE_SYSTEM_SPEC.md`](OFFICIAL_COORDINATE_SYSTEM_SPEC.md)、[`OFFICIAL_TCP_COMMAND_REFERENCE.md`](OFFICIAL_TCP_COMMAND_REFERENCE.md)。

> ⚠️ **重要前提**:來源 PDF 是 **DobotStudio Pro 的控制器內 Lua 腳本應用**(`Go`/`Move`/`RP`/`TCPRead`/`WaitDI`/`"SYNC=1"`),**不是 TCP/IP 遠端接口規格**。本筆記的價值在於**校正方法與設計概念可以萃取通用**;凡涉及實際下指令,一律對應到 TCP/IP 30003/29999 介面(見 §5 對照表)。

## 1. 校正方法萃取(忠於手冊)

這份 DEMO 同時做了**兩套互相獨立的校正**:

### 1.1 工具坐標系 — 兩點示教法
- 末端裝**偏心吸嘴** + 標定針。因為吸嘴偏離法蘭軸心,TCP(Tool Center Point)不在法蘭中心,必須標定。
- 方法:TCP 以**兩種不同姿態**對準空間中**同一參考點**,各「獲取坐標」(P1、P2),系統由此求出工具的位置偏移,生成工具坐標系。
- 路徑:DobotStudio Pro →「設置 > 坐標系管理 > 工具坐標系」→ 選坐標系 →「修改 > 兩點設置」。
- **DEMO 全程使用 Tool = 1**。

### 1.2 視覺標定 — N 點(4 點)標定資料流
建立「視覺坐標系 ↔ 機械坐標系」的轉換,流程:

1. VisionStudio「4-points cali」工程拍標定紙,辨識 4 個十字 → 得 **ID0~ID3 的像素座標**(**順序至關重要**,第一點 ID=0)。
2. 在 DobotStudio Pro,把標定針依序移到 ID0~ID3,用「腳本編程 > 存點」讀出這 4 點的**機械 (x, y)** 座標。
3. 把 4 組機械 (x, y) 填回 VisionStudio 標定點編輯界面的「物理坐標 X / Y」→ 存成**標定檔**。
4. VisionStudio「Calib Transformation」模組載入標定檔 → 之後視覺輸出的就**直接是機械坐標系下的 (x, y, r)**。

→ 關鍵:**坐標轉換在視覺端完成**,送給機械臂的已是機械座標;機械臂不自己做仿射變換。
→ 通訊:視覺端開 **TCP server**(DEMO:IP `192.168.1.40`, port `4001`),機械臂當 client 連入,收逗號字串 `"x,y,r"`。

## 2. 與 ArUco 的等價對應

手冊的「N 點標定」在數學上就是**平面 影像→機械臂 坐標映射**,與 ArUco 定位是同一件事;ArUco 只是把「找對應點」自動化、變穩。

| 步驟 | 手冊(十字標定紙) | 本專案 ArUco |
|---|---|---|
| 影像側對應點 | 人工找 4 個十字中心像素,**順序要對(ID0~ID3)** | marker 角點/中心,**ID 自帶,免排序** |
| 機械側對應點 | 標定針逐點觸碰讀機械 (x,y) | 一樣要取得:觸碰 or 已知治具位置 |
| 解出的變換 | 像素 → 機械座標 | 同一個變換(仿射 / 單應) |
| 變換套用處 | VisionStudio「Calib Transformation」 | 本專案主機端(Python) |

### 數學
平面取放只需 **2D 仿射(6 參)**,鏡頭斜/有透視才需 **單應(8 參)**:

```
[x_robot]   [a b c]   [u]
[y_robot] = [d e f] · [v]
                       [1]
```

仿射 3 點即可解,**取 4 點做最小二乘更穩**(這正是手冊取 4 點的原因)。對應 OpenCV `estimateAffine2D` / `findHomography`(本筆記不附程式,僅標觀念)。

### ArUco 比十字法多賺到的
- **免排序**:手冊最脆弱處是「標定點順序非常重要,錯一個全錯」;ArUco 的 ID 直接解決。
- **r 直接得**:手冊的視覺要另算 r 回傳;ArUco 由 marker 角點即可算出影像內角度 → 加偏移得機械臂 r。
- **可單板標定**:一塊多 marker 板一次給多組對應點。

## 3. 兩個校正別混(設計時最容易踩的坑)

| 校正 | 解的是什麼 | 跟相機有關嗎 | 何時重做 |
|---|---|---|---|
| **工具坐標系**(兩點示教,Tool=1) | 法蘭 → TCP 偏移 | 無關,是夾具本身 | 換夾具/吸嘴 |
| **視覺標定**(4 點 / ArUco) | 相機像素 → 機械平面 | 有關 | 換相機/移動相機/重貼板 |

兩者是**串接**的:視覺告訴你「物件在機械臂座標哪裡」,Tool=1 保證「偏心吸嘴實際對得到那個點」。
→ 介面設計上要讓「換相機只重標視覺、Tool 不動」這件事很自然(各校正獨立、各自有 artifact)。

## 4. 七條坐標系介面設計概念(核心)

每條:**概念 → 手冊哪裡體現 → 你介面該怎麼設計**。

### ① 坐標系是「可組合的變換鏈」,每層各自標定
- 體現:一個取放點走 `相機像素 →(視覺標定)→ 機械臂 base/user →(Tool 標定)→ 偏心吸嘴 TCP`。
- 設計:把 frame 建模成一串可組合變換(`T_base←user`、`T_flange←tool`、`T_robot←cam`),每段獨立、可單獨重標。換一段不影響其他段。

### ② 逐指令顯式綁 User/Tool,勝過隱式全域模式
- 體現:每個點都帶 `{coordinate, tool=1, user=0}`,而非靠「先設全域坐標系」的隱藏狀態。
- 設計:運動 API 讓**每筆**都能宣告表達在哪個 User/Tool(對應 `MovL(...,User=,Tool=)`)。顯式綁定 = 無狀態、可重入、好 debug;全域模式容易「忘了現在是哪個系」。

### ③ 標定 = 「對應點 + 求解器 + 存成 artifact」的通用抽象
- 體現:Tool 標定(2 姿態→偏移)與視覺標定(4 點→仿射)其實同一抽象。
- 設計:做一個通用 `Calibration`:加對應點 → 求解 → 存/載入。**別把標定數字寫死**;標定檔要可版本化、可追溯(誰、何時、用哪些點標的)。

### ④ 轉換在主機端做完,機械臂介面只收「絕對座標」
- 體現:手冊刻意把仿射放 VisionStudio,**送機械臂的已是機械座標**。
- 設計:讓機械臂介面保持「薄而笨」——只接受某宣告 frame 下的絕對座標並執行;所有 transform 邏輯留主機。介面職責單一,出錯易二分(算錯 vs 動錯)。

### ⑤ 點位模型要能「部分來自感測、部分固定/示教」
- 體現:視覺只給 `x, y, r`;`z` 固定(抓取 -13.5、接近 +10.5)。
- 設計:point 不是死的四元組,而是**模板**——某些軸綁感測輸入、某些軸是常數/示教值。同一取放動作可重用於不同物件。

### ⑥ 基準點 + 參數化偏移,取代逐點示教
- 體現:碼垛用 `RP(P4, {40·col, 40·row, ...})` —— 一個示教錨點 + 程式生成格點(**格距 40mm**)。
- 設計:支援「錨點 + 偏移產生器」,坐標由公式生成而非逐點教;介面提供 grid/pattern 產生器(輸入錨點與間距)。

### ⑦ 用「身分」而非「順序」;統一單位與位姿格式
- 體現:手冊最脆弱處是點序;ArUco 用 ID 免疫。
- 設計:凡引用 frame / marker / 點位,一律用**穩定 ID** 而非位置順序;全介面統一一種位姿表示(`{x, y, z, r}`,mm / 度),杜絕單位與順序的隱性 bug。

## 5. 腳本 API → TCP/IP 對照表(實際下指令時用這個)

來源 PDF 的 `Go/Move/RP/...` 是控制器內 Lua,**不是** 30003/29999 遠端指令。映射:

| 腳本(本 PDF) | TCP/IP 遠端等效 | 註記 |
|---|---|---|
| `Go(pt)`(PTP/關節) | `MovJ(x,y,z,r,User=0,Tool=1)` | `tool/user` → `Tool=/User=` 可選參數 |
| `Move(pt)`(直線) | `MovL(x,y,z,r,User=0,Tool=1)` | 同上 |
| `{coordinate={x,y,z,r}, tool=1, user=0}` | `MovL(...,User=0,Tool=1)` | **逐點綁定**坐標系,與遠端可選參數一致 |
| `RP(P4,{dx,dy,dz,dr})` | 主機自行算 `P4+offset` 後 `MovL` | RP 相對「某示教點」;`RelMov*User` 相對「當前位置」,語意不同,**建議主機端算絕對座標** |
| `Go(...,"SYNC=1")` | **改用 `Sync()` / `SyncAll()`** | 官方 §3.1:**TCP 運動指令不支援 `SYNC` 參數**(見 `OFFICIAL_TCP_COMMAND_REFERENCE.md`) |
| `WaitDI(n,1)` | 輪詢 `DI(n)` 直到回 1 | |
| `DO(n,s)` | `DO(index,status)` | 一致 |

> DEMO 的 I/O 慣例(供對照):DI2 啟動 / DI3 停止 / DI4 復位 / DI16 急停;DO10 吸真空 / DO11 破真空 / DO6~8 三色燈。

## 6. 一句總結

把「坐標系」當**一級公民**:**可組合、各自標定、逐指令顯式綁、轉換在邊緣做、機械臂端只執行絕對座標**。
照這幾條設計介面,視覺(ArUco)接上來幾乎即插即用 —— 視覺/主機算出機械座標 `(x,y,r)`、補固定 `z`、帶 `Tool=1` 下發 `MovL`,機械臂端不做任何仿射。
